## How SSM Parameters Work

### Three Loading Patterns

SSM parameters in this platform are consumed via three distinct patterns:

**Pattern 1 — CDK Deploy-Time Reads:**
```csharp
// CdkStackUtilities, ServerlessApiStackHelper, GatewayStack, etc.
StringParameter.ValueForStringParameter(stack, $"/{env}/tp/SUBNET_1");
StringParameter.ValueFromLookup(stack, $"/{env}/tp/VPC_NAME");
```
These are evaluated during `cdk synth`. If the parameter doesn't exist, the CDK deploy **fails immediately**.

**Pattern 2 — Runtime `ParameterStoreHelper` (most services):**
```csharp
// LambdaEntry.cs / Function.cs in 20+ services
ParameterStoreHelper.LoadParametersToEnvironmentAsync($"/{env}/tp/InternalServices");
```
Reads all parameters under a path recursively via `GetParametersByPathAsync`. Converts the last path segment to an environment variable name. Runs at Lambda cold start.

**Pattern 3 — Runtime `ReadSsmParametersAndAddToEnvVars` (4 services):**
```csharp
// CSV generator, PDF generator, bandsintown, marketing-feeds
ReadSsmParametersAndAddToEnvVars(env);
// Reads from /{env}/tp/{service-name}/ with Recursive=true, WithDecryption=true
```
Same mechanism as Pattern 2 but implemented inline in each service's `Function.cs` rather than via the shared helper. Reads ALL parameters under a service-specific SSM path prefix.

### Two Sources of Environment Variables

Each Lambda gets env vars from two sources:
1. **CDK env-var JSON** — baked into Lambda configuration at deploy time (static)
2. **SSM Parameter Store** — loaded at runtime during cold start (dynamic)

Keys from CDK do NOT need to be in SSM. The tables below list **only keys that must come from SSM**.
